## 3.1 Робость в роутере services: `get_studio` → `get_studio_or_raise`

**Цель:** Единообразно получать студию во всех эндпоинтах services; при отсутствии студии — явная 404, а не падение на `None` в `ensure_studio_owner`.

**Файлы:** `backend/app/api/v1/services.py`

**Задачи:**
1. Заменить все вызовы `get_studio(uow, ...)` на `get_studio_or_raise(uow, ...)` в эндпоинтах:
   - `create_service_endpoint` (schema.studio_id)
   - `update_service_endpoint` (service.studio_id)
   - `deactivate_service_endpoint` (service.studio_id)
   - `create_service_schedule_endpoint` (service.studio_id)
   - `delete_schedule_endpoint` (service.studio_id)
2. Обновить импорт: `from app.services.studio import ensure_studio_owner, get_studio_or_raise` (убрать `get_studio`).
3. **Попутно:** добавить отсутствующий импорт `delete_schedule` в блок импортов из `app.services.service` (сейчас используется в `delete_schedule_endpoint`, но не импортирован).

**Критерий готовности:** В роутере services нигде не используется `get_studio`; при несуществующей студии возвращается 404 с сообщением «Студия не найдена».

## 3.2 Политика времени (naive UTC) и устранение дублирования

**Цель:** Одна точка конвертации времени (UTC), единая политика для БД и сравнений; убрать дублирование `_to_naive_utc` в трёх местах.

**Файлы:**
- `backend/app/core/repositories/slot_repo.py`
- `backend/app/services/slot.py`
- `backend/app/services/service.py`

**Задачи:**
1. **Вынести утилиту в одно место.** Варианты:
   - **Вариант A:** `app/core/datetime_utils.py` — функция `to_naive_utc(dt: datetime) -> datetime` и краткий комментарий о политике: «Все времена в БД и сравнениях — naive UTC».
   - **Вариант B:** оставить в `slot_repo` и импортировать в `slot` и `service` сервисы из репозитория (минус: сервисный слой зависит от деталей репозитория).
   - **Рекомендация:** Вариант A — нейтральный модуль в `core`, от него зависят и репозиторий, и сервисы.
2. В `slot_repo`, `services/slot.py`, `services/service.py` удалить локальные `_to_naive_utc` и использовать общую функцию из `core.datetime_utils`.
3. В `docs` или в docstring модуля зафиксировать: «Время слотов и расписаний хранится и сравнивается в UTC (naive datetime в Python); ввод от API конвертируется в UTC при сохранении.»

**Критерий готовности:** Одна реализация `to_naive_utc` в core; все сравнения с `Slot.start_time`/`end_time` идут через неё; дублирования нет.

## 3.3 Роль `get_db` в deps

**Цель:** Явно определить назначение `get_db`: оставить для возможных read-only сценариев без UoW или убрать из публичного API.

**Файлы:** `backend/app/api/deps.py`, при необходимости `backend/app/core/database.py`

**Задачи:**
1. **Вариант A (оставить):** В `deps.py` в `__all__` оставить `get_db`; добавить комментарий над объявлением или в docstring модуля: «Экспортируется для возможных read-only эндпоинтов без UoW; роутеры по умолчанию используют get_uow.»
2. **Вариант B (убрать из deps):** Удалить импорт и реэкспорт `get_db` из `deps.py`; при необходимости использовать `get_db` импортировать напрямую из `app.core.database`. Тогда в deps остаются только зависимости, используемые роутерами.

**Рекомендация:** Вариант A — минимальное изменение, явный контракт без удаления возможности использовать сессию без UoW в будущем.

**Критерий готовности:** В коде и/или документации зафиксировано, что роутеры используют `get_uow`, а `get_db` предназначен для редких read-only случаев.

## 3.5 Интеграционные тесты webhook с реальной БД

**Цель:** 1–2 интеграционных теста, где создаётся бронирование/заказ, вызывается webhook с корректной подписью Stripe и проверяется смена статуса в БД (с использованием `client` из conftest и rollback).

**Файлы:** `backend/tests/test_webhooks.py`, при необходимости `backend/tests/conftest.py`

**Задачи:**
1. Добавить фикстуру или хелпер для подписи Stripe (например, воспроизводимый секрет для тестов и расчёт `stripe_signature` для payload).
2. Интеграционный тест «booking»: создать студию/услугу/слот, создать бронирование в статусе pending, вызвать POST webhook с событием `checkout.session.completed` и metadata `booking_id`; проверить, что статус бронирования обновился (и при необходимости что слот помечен занятым и т.д.).
3. Интеграционный тест «order»: по аналогии — создать заказ, вызвать webhook с `order_id` в metadata, проверить смену статуса заказа в БД.
4. Использовать `client` из conftest (app_with_rollback_uow), чтобы после теста откатывать транзакцию и не засорять БД.

**Критерий готовности:** В CI проходят минимум один интеграционный тест для booking и один для order через реальный HTTP-вызов webhook и проверку состояния в БД.

## 3.6 Рейт-лимит на чувствительные write-операции

**Цель:** Снизить риск злоупотреблений: добавить лимиты на создание бронирований и создание checkout-сессий (по IP или по user_id после логина).

**Файлы:** `backend/app/main.py`, `backend/app/core/rate_limit.py` (или отдельный модуль лимитов), роутеры `bookings`, `payments`

**Задачи:**
1. Определить лимиты (например: создание бронирования — N запросов в минуту на IP; создание checkout-session — M в минуту на IP или на user_id). Зафиксировать значения в конфиге или константах.
2. Подключить `limiter` к эндпоинтам:
   - `POST /bookings` (create_booking_endpoint),
   - `POST /payments/checkout-session` и `POST /payments/order-checkout-session`.
3. При необходимости добавить ключ по `user_id` для аутентифицированных запросов (например, лимит «X запросов в минуту на пользователя» в дополнение к IP).
4. Убедиться, что при срабатывании лимита возвращается 429 и обрабатывается существующим обработчиком `RateLimitExceeded`.

**Критерий готовности:** На указанных эндпоинтах действуют лимиты; при превышении возвращается 429; в документации или конфиге указаны значения лимитов.

## 3.7 Логирование 4xx/5xx в middleware

**Цель:** Не логировать успешные 4xx (например, 404) как предупреждение/ошибку; для 5xx и критичных 4xx (например, 401) усилить уровень и контекст.

**Файлы:** `backend/app/core/middleware/logging_middleware.py`, при необходимости `backend/app/main.py` (exception handler).

**Задачи:**
1. В `RequestLoggingMiddleware` после получения `status_code`:
   - Для 2xx и «нормальных» 4xx (например, 404, 422) оставить текущий уровень `logger.info` или понизить до `debug` для части 4xx — по решению команды.
   - Для 5xx и критичных 4xx (401, 403, 429) логировать с уровнем `logger.warning` или `logger.error` (5xx) с теми же полями (method, path, status, duration_ms, request_id, user_id).
2. Опционально: в exception handler для необработанных исключений добавлять более подробный контекст (например, тип исключения, traceback в лог при 5xx).

**Критерий готовности:** В логах чётко разделены обычные запросы и проблемные (5xx, 401, 403, 429); при необходимости 404 не засоряют уровень warning.
